## Overall Impression

Solid foundation with real discipline — the color math, terminal-native thinking, and restraint are the genuine strengths here. The document knows what it's doing and why. The gaps are structural: undo is missing entirely, the CLI path (the primary interaction mode per PRODUCT.md) isn't designed, and error states are assumed to never happen. The accessibility section asserts intent without specifying implementation — which is fine for a v1 spec but needs to be resolved before build.

## What's Working

1. **Color math is rigorous.** OKLCH throughout, verified contrast ratios at AAA/AA, chroma sanity checks, explicit hue separation between primary and accent. Professional-grade color work.
2. **Terminal-native discipline.** Instant transitions, respects terminal font prefs, no pixel-based padding, no web-in-terminal cargo-culting. The "reduced motion: all instant by design" decision is elegant — it sidesteps the whole complexity instead of engineering around it.
3. **Coherent restraint.** 6 colors for the palette, 5 components, one layout row plus a footer. Nothing unnecessary. The "out of your way" principle is baked into every decision.

## Priority Issues

### P0: No undo / error recovery

Regenerating a palette overwrites the current one. No undo, no history, no "previous" navigation. A single wrong keystroke destroys work.

- **What:** Missing undo/history model
- **Why:** The spec lets one keystroke destroy unsaved work. Terminal users expect undo (Cmd+Z / Ctrl+Z is muscle memory)
- **Fix:** Add a regeneration log (last N palettes) accessible via `[`/`]` or an explicit undo keybinding. Configure history size via config.
- **Suggested command:** `$impeccable harden DESIGN.md`

### P0: Only the TUI is designed; CLI mode is absent

PRODUCT.md describes `palette generate --mood dark --count 5` as the primary value prop. DESIGN.md only covers the TUI. The CLI is the entry point and likely the more-used path for many users.

- **What:** Missing CLI interaction design
- **Why:** The spec doesn't describe what the CLI output looks like, what flags control formatting, how piping works, or what `--help` renders
- **Fix:** Add a "CLI Interface" section parallel to "TUI Components" describing output layouts, flag behavior, pipe support, and help formatting
- **Suggested command:** `$impeccable shape cli-interface`

### P1: No empty / null / error states

What does the TUI show on first launch? What happens when color generation fails? What about terminal resize?

- **What:** Missing state coverage
- **Why:** The first impression is an empty state — it sets expectations. Error handling determines trust.
- **Fix:** Add a "States" subsection: initial/empty, loading, error, terminal resize behavior
- **Suggested command:** `$impeccable harden DESIGN.md`

### P1: Screen reader support is asserted but not specified

"Screen-reader compatible output structure" is listed but no implementation detail exists. Screen reader support in a TUI is non-trivial (requires aria roles, live regions, or OSC sequences).

- **What:** Accessibility assertion without design detail
- **Why:** An assertion without a design is a gap that will be found during implementation
- **Fix:** Specify the accessibility approach per component (e.g., aria-live regions, how state changes are announced, focus management)
- **Suggested command:** `$impeccable harden DESIGN.md`

### P2: Key binding help overlay is underspecified

Listed as "overlay listing shortcuts, grouped by category" — no layout, no key names, no navigation model for the overlay itself.

- **What:** Thin spec for an interactive element
- **Why:** The help overlay is the user's safety net. Ambiguity creates a discoverability gap.
- **Fix:** Expand the spec: overlay dimensions, navigation model, dismissal behavior, example layout
- **Suggested command:** `$impeccable shape help-overlay`

## Persona Red Flags

**Alex (Power User):** No undo is the biggest red flag. Alex expects Ctrl+Z to work. Regeneration has no guardrails. No batch operations or piped scripting described. Will abandon on the first accidental overwrite.

**Jordan (First-Timer):** No empty state means Jordan sees a blank screen on first launch with no guidance. No in-app help text (beyond the key binding overlay) means Jordan has to guess what to do. No `--help` design means Jordan's first CLI interaction might be confusing.

**Sam (Accessibility-Dependent):** Screen reader support is asserted but unspecified. No detail on focus indicators, announcement of state changes, or how the TUI renders for speech. Sam will discover the gaps, not the documentation.

## Minor Observations

- `muted` token name describes a visual attribute, not a semantic role. `secondary` or `label` would be clearer.
- No mention of persisted preferences (default mood, count, export format) in the TUI design.
- Color export targets (CSS variables, Tailwind) from PRODUCT.md are not detailed in DESIGN.md.
- No spec for what happens when terminal width is too narrow for all swatches.

## Questions to Consider

- You describe "mood-driven" palette generation — but DESIGN.md never defines what "mood" means as a color input. Are moods a fixed set or free text? How does the user choose one?
- Can a user navigate backward in a TUI session? If I generate, lock two, regenerate, then realize the first was better — how do I get it back?
- Is "5 colors" the only count, or can the user request 3 or 8? How does the layout scale beyond terminal width?
